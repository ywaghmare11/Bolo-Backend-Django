## 2026-07-23 (3)

- `[BE]` `[INFRA]` `feature/roadmap-hardening`, Stage B of 2 ‚Äî the remaining two deferred Phase 2/3 items: `HasOrgRole`/`IsTenantMember` permission classes and generic audit logging (CLAUDE.md Architecture Rules point 8).
  - **`HasOrgRole`/`IsTenantMember`** (`apps/common/permissions.py`): `IsTenantMember` checks `request.tenant_id` truthy (an explicit marker for the "every view is tenant-scoped" rule ‚Äî repositories still do their own tenant_id filtering regardless, this doesn't replace that). `HasOrgRole(allowed_role_levels)` is a permission-class *factory* (mirrors the original Node backend's `requireOrgRole(['TOP'])` middleware) checking `request.role_level` against the given list. Wired into a real endpoint rather than left standalone: `GET /api/v1/tenant/` (`docs/api/api-spec.md`'s already-documented `requireOrgRole(['TOP'])` overview endpoint ‚Äî nothing invented, just the one piece of that endpoint's contract, `memberCount`/`deptCount`/etc, not the full `apps/tenants` surface). New `apps/tenants/{repositories,services,serializers,views,urls}.py` + `apps/tenants/tests/` (converted from the unused `tests.py` stub to a `tests/` package, matching every other app). `TenantRepository.get_with_counts` does three plain queries rather than one annotated one ‚Äî `TenantMembership`'s composite PK `(tenant, user)` makes `Count(..., distinct=True)` raise when combined with a second to-many `Count` annotation in the same query; not an issue for a single-row admin lookup with no N+1 risk.
  - **Audit logging infra:** `celery==5.6.3` + Celery settings added (`config/celery.py`, `config/__init__.py` exports `celery_app`, `CELERY_BROKER_URL`/`CELERY_RESULT_BACKEND` reuse the same `REDIS_URL` as the throttle's cache). Hit a real local compatibility snag: `redis` 8.x defaults its connection protocol to RESP3 (sends a `HELLO` handshake), which only Redis 6+ speaks -- this machine's local Redis is 5.0.7. django-redis lets you override the protocol via `CACHES.OPTIONS`, but Celery's broker doesn't (kombu parses the broker URL itself and has no `protocol` passthrough at all ‚Äî confirmed by tracing the `TypeError` down to `kombu.connection.Connection._init_params`). Pinned `redis==5.2.1` instead (RESP2-only, no handshake, works against both old and new Redis) rather than patching around it. `config/settings/test.py` sets `CELERY_TASK_ALWAYS_EAGER`/`CELERY_TASK_EAGER_PROPAGATES` so tests run tasks synchronously with no worker needed.
  - **Middleware + route config** (`apps/common/audit_middleware.py`, `apps/common/audit_route_config.py`, `apps/common/tasks.py`): generic, not dispatched ‚Äî no view/service calls it. The middleware resolves the URL itself (`django.urls.resolve`) rather than reading `request.tenant_id`/`request.user`, because those are set on DRF's internal `Request` wrapper created inside `APIView.dispatch()`, not on the underlying `HttpRequest` a plain Django middleware holds a reference to (confirmed by tracing DRF's `Request` class ‚Äî no `__setattr__` override, so attribute writes during authentication never propagate back out). Instead it independently decodes the same httpOnly access-token cookie `CookieJWTAuthentication` uses, purely to identify the actor/tenant. Route config covers every currently-mutating `apps/tasks` route (create/update/delete/accept/done-a/done-d/cancel ‚Äî `resolve_task_update_action` diffs before/after for the multi-purpose `PATCH`, same priority order as `TaskService.update_task`'s own field handling: reassign > label > due date > priority) plus the one documented exception, `USER_LOGIN`/`USER_LOGOUT` off `User.last_login_at`/`last_logout_at`. `task-remind` and all of `apps/labels` are deliberately excluded ‚Äî remind doesn't mutate a persisted entity, and no `AuditAction` enum values exist for label events (didn't invent one, same principle as the Notifications rule about adding the event type first). Writes are dispatched via `write_audit_log_task.delay(...)` (fire-and-forget, only when `response.status_code < 400`).
  - **Tests:** `apps/common/tests/test_audit_middleware.py` (new `tests/` package, same conversion as tenants) ‚Äî 9 tests covering `TASK_CREATED`/`TASK_REASSIGNED`/`TASK_DUE_DATE_CHANGED`/`TASK_DELETED` (null `after`)/`TASK_STATUS_CHANGED`, a 403 that must **not** write a row, and login/logout. Also ran a real end-to-end smoke test outside pytest ‚Äî a live `celery worker` process against the real (now-pinned) Redis, `manage.py runserver`, and a `curl` POST ‚Äî confirmed the task is actually enqueued, picked up by the worker, and the `AuditLog` row lands, not just the eager-mode test path. Full suite: 37 tests, all green; `ruff check` clean on every new/edited file (pre-existing findings in untouched `apps/common/{admin,views}.py` scaffold stubs left alone, same as prior entries).

## 2026-07-23 (2)

- `[BE]` `[INFRA]` `feature/roadmap-hardening`, Stage A of 2 ‚Äî three of ROADMAP.md's own deferred Phase 2/3 checklist items (see 2026-07-19 entry below for why they were deferred, not oversights): (1) Redis-backed OTP-request throttle, (2) Task indexes, (3) a query-count regression test. Item 2 (HasOrgRole/IsTenantMember + audit logging) is a separate Stage B commit ‚Äî bigger diff, needs Celery/Redis infra and a route-config table touching most existing views.
  - **Redis throttle:** `redis`/`django-redis`/`celery` added to `requirements/base.txt` (celery installed now, wired up in Stage B). `CACHES["default"]` in `config/settings/base.py` now points at `REDIS_URL` via `django_redis` (this is also the shared cache DRF's `ScopedRateThrottle` reads/writes through ‚Äî there's no separate throttle-specific cache setting). `RequestOtpView` (`apps/auth/views.py`) gained `throttle_classes=[ScopedRateThrottle]`, `throttle_scope="otp_request"`, rate `5/min` in `REST_FRAMEWORK.DEFAULT_THROTTLE_RATES` ‚Äî layered on top of, not replacing, `AuthService.request_otp`'s existing per-email 60s DB-timestamp cooldown (that check alone doesn't stop someone cycling many different emails from one IP; this one is IP-keyed and Redis-backed so it holds across multiple gunicorn workers, unlike an in-memory cache). Needed a fix to `config/exception_handler.py` ‚Äî DRF's default `Throttled` response is a bare `{"detail": "..."}`, not this project's `{success, error:{code,message}}` envelope, so it's now translated to a `429 RATE_LIMITED` like the existing OTP lockout responses. `config/settings/test.py` overrides `CACHES` to `LocMemCache` (tests shouldn't need a running Redis) and a new root `conftest.py` clears the cache before/after every test (the throttle's counters live in the cache, which ‚Äî unlike the DB ‚Äî Django's test runner doesn't reset between tests on its own; without this, unrelated tests calling `request-otp/` more than a handful of times across a session would start tripping the 5/min limit). `.env`/`.env.example`'s existing `REDIS_URL` is now actually consumed.
  - **Indexes:** new migration `apps/tasks/migrations/0004_...` adds a composite `(tenant_id, status)` index (every `TaskRepository` list query filters both), single-column indexes on `assignee_id` and `due_date`, and a partial index on `is_archived=False` (Postgres `WHERE` clause ‚Äî list views default to unarchived, keeps the index small). Each `AddIndex` operation has an inline comment naming the query it serves, per the task's ask; the pre-existing `idx_tasks_status` was left in place (not asked to remove it, still cheap).
  - **Query-count regression test:** turned out `apps/tasks/repositories.py` already had no N+1 to fix ‚Äî `_annotated_queryset`'s `select_related`+`annotate(Count(...))` and `attach_latest_comments`'s single bulk query already avoid it (ROADMAP.md Phase 3's `prefetch_related` wording doesn't literally apply; counts are done via `annotate` instead). New `apps/tasks/tests/test_query_optimization.py` (+ a new minimal `apps/tasks/factories.py:TaskFactory`, didn't exist before) asserts the list endpoint holds at exactly 4 queries with 3 tasks and again with 8 ‚Äî proving the count doesn't scale with row count, which is what would catch a reintroduced N+1.
  - Ran the full suite (24 tests, including 2 new ones) ‚Äî all green. `ruff check` clean on every new/edited file.
